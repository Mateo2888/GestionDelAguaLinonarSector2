## Qué es esto
Sitio web comunitario para gestionar el sistema de agua del barrio Limonar Sector II, Soacha, Cundinamarca. Es un sitio estático (un solo `index.html` con CSS y JS embebidos) conectado a Supabase como backend.

- Publicado con GitHub Pages en: https://mateo2888.github.io/GestionDelAguaLinonarSector2/
- Archivo principal: `index.html` en la raíz del repo (single-file: HTML + `<style>` + `<script>`)
- Otros archivos en el repo: `supabase-setup.sql` (script de creación de tablas), `INSTRUCCIONES.md` (guía de puesta en marcha)

## Stack técnico
- HTML5 + CSS3 (variables CSS, glassmorphism, modo oscuro vía `data-theme` en `<body>`) + JavaScript vanilla (sin frameworks ni build step)
- Supabase JS v2 cargado por CDN (`https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2`) para: base de datos (Postgres), autenticación (login del panel admin), y storage (fotos de reportes de daño)
- Font Awesome 6.5.1 (CDN) para íconos, Google Fonts (Poppins + Inter)
- Sin backend propio, sin Node/build tools — todo corre en el navegador

## Arquitectura del JS en index.html (para orientarte en el código)
- Bloque de config arriba del `<script>` principal: `SUPABASE_URL`, `SUPABASE_ANON_KEY`, variable `sbClient` (cliente de Supabase) y flag `usingSupabase`.
- Objeto `DEMO` con datos de ejemplo (fallback si Supabase no está configurado) y `STATE` (estado en memoria que se renderiza).
- `loadAll()` trae datos de las 8 tablas de Supabase y llama a las funciones `render*()` de cada sección (programación, noticias, galería, mantenimientos, finanzas, documentos).

### 2026-07-21 — Ahorro de almacenamiento de Supabase (borrar archivos huérfanos + comprimir fotos)
**Qué se hizo:** el usuario preguntó qué medidas se podían implementar para no superar nunca el plan gratuito de Supabase (500 MB de base de datos, 1 GB de Storage). Primero se le explicó que la mayoría del sitio (noticias, galería, mantenimientos) ya usa URLs de imagen en vez de subir archivos, así que el único consumo real de Storage son 2 formularios: "Reportar daño" (foto, público) y el comprobante de Finanzas (admin). Se detectó y corrigió un problema concreto, más una mejora preventiva:
- **Fuga de almacenamiento corregida:** al borrar un reporte de daño o un movimiento de finanzas, la foto/comprobante subido a Supabase Storage NO se borraba junto con el registro — quedaba huérfano ocupando espacio para siempre. Se agregó `deleteStorageFile()` (convierte la URL pública de vuelta a su ruta interna con `storagePathFromUrl()` y la borra del bucket `fotos`) y se conectó a `crudDelete()` mediante un mapa `STORAGE_FIELDS` (`reportes_dano→foto`, `finanzas→comprobante`), así que ahora borrar el registro también libera el archivo. También se corrigió el mismo problema al **editar** un movimiento de finanzas y reemplazar su comprobante por uno nuevo: antes el comprobante viejo se quedaba huérfano; ahora se borra en cuanto el nuevo se sube con éxito.
- **Compresión de fotos en el navegador antes de subir:** se agregó `compressImage()`, que redimensiona la foto a un ancho máximo de 1600px y la recomprime a JPEG calidad 82% usando un `<canvas>`, todo dentro del navegador de quien sube el archivo (sin tocar el servidor). Una foto de celular típica de 3-5 MB queda en unos cientos de KB, sin pérdida visible en pantalla — con esto el 1 GB gratuito rinde para muchas más fotos. Se aplicó a los 2 únicos formularios que suben archivos reales (foto de "Reportar daño" y comprobante de Finanzas). Si la compresión falla o el resultado queda más pesado que el original (raro, pero posible con imágenes ya muy comprimidas), se sube el archivo original sin cambios — nunca se sube algo peor de lo que el usuario ya tenía.
Se validó exhaustivamente: sintaxis JS, balance de etiquetas, cero ids duplicados, cada `onclick`/`.then()`/`getElementById` contra su definición real, y se probó `storagePathFromUrl()` de forma aislada contra una URL real de ejemplo para confirmar que extrae la ruta correctamente (incluyendo nombres de archivo con espacios/caracteres especiales codificados).
**Qué quedó pendiente:** no hay forma de probar visualmente que una foto real se comprime y se sube bien en este entorno (no hay navegador) — pedirle al usuario que suba una foto de prueba desde su celular en "Reportar daño" y confirme que se ve bien, y opcionalmente que revise en Supabase (Storage > fotos) que el archivo subido pesa notablemente menos que el original. Medidas adicionales que se mencionaron pero no se implementaron por quedar fuera de lo pedido: limitar la longitud de texto en los formularios públicos, y borrar periódicamente reportes/afiliaciones ya resueltos (housekeeping manual del administrador).
